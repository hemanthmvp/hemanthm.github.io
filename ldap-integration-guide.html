What is Active Directory & Why Does ServiceNow Need It?

When you join a company, the IT team creates your profile — name, email, department, role — on a central server. ServiceNow also needs this same data to assign you tickets, give you access, and place you in the right team.

02

What is LDAP?

LDAP stands for Lightweight Directory Access Protocol. It is the language/protocol that ServiceNow uses to talk to your Active Directory and fetch user and group information from it.

📚 Analogy — Think of it this way
Your company's Active Directory is like a huge library. Every user record (name, email, department, role) is a book stored on different shelves.

ServiceNow wants to read those books — but to do that, it needs to know: how to enter the library, where the shelves are, and how to search for the right book.

That set of rules — the language ServiceNow uses to communicate with the directory — is called LDAP.

LDAP Ports

389
Standard LDAPDefault port for unencrypted LDAP connections. Used in development & testing.
STANDARD
636
LDAPS (Secure)SSL-encrypted LDAP. Always use this in production environments.
SECURE
⚠ Production Rule
In a real organisation, your AD/IT team will give you the server URL, port, login DN, and password. Always enable SSL (port 636) in production. Never use port 389 in live environments.
03

Understanding LDAP Structure: DC · OU · CN

This is what confuses every beginner. LDAP is not a flat list of users — it's a tree structure, just like your company's org chart. Once you understand these 3 terms, everything else clicks.

DC
Domain Component
ROOT OF THE TREE
The name of your organisation's domain. The very top of the LDAP tree. Every domain is split into parts separated by dots.
glidescript.com →
DC=glidescript, DC=com
OU
Organisational Unit
DEPARTMENT / FOLDER
Think of these as departments or folders inside your org — HR, Engineering, Finance, IT, Support. OUs sit below DC and group users together.
OU=finance → folder
containing all finance users
CN
Common Name
INDIVIDUAL OBJECT
The natural name of a specific user or group. This is the leaf node — the actual person or group object at the bottom of the tree.
CN=Anjeli Sharma
CN=IT-Team (a group)
LDAP Tree Structure — Visual
DC=glidescript, DC=com← Root (domain)
│
├── OU=employees← Department folder
│   │
│   ├── OU=it← Sub-department
│   │   │
│   │   └── CN=Anjeli Sharma← Individual user
│
└── OU=finance← Another department
    └── CN=Rahul Sharma← Finance user
Full Distinguished Name (DN) — Read RIGHT to LEFT
CN=Anjeli Sharma, OU=employees, OU=it, DC=glidescript, DC=com
Reading it: domain glidescript.com → go into IT dept → employees → find user Anjeli Sharma

Step-by-Step: Creating the LDAP Server in ServiceNow

We are using the free online LDAP test server at ldap.forumsys.com — the only publicly available LDAP server for testing. It has real users, real groups, and real data.

🔍 Finding the Test Server
Search Google for "LDAP forumsys" → Open the first result (Online LDAP Test Server). You will find the server URL, login DN, and password there. In a real organisation, your IT/AD team provides these credentials.
1
Navigate to LDAP in ServiceNow
NAVIGATION
In ServiceNow, you create LDAP server records under the System LDAP application. This introduces the external Active Directory to ServiceNow.
How to do it
  1. Go to the Filter Navigator (top-left search bar)
  2. Type ldap
  3. Under System LDAP, click New Server
  4. Or click LDAP Servers to see existing ones
2
Fill in the LDAP Server Form
CONFIGURATION
This form is your way of introducing the LDAP server to ServiceNow — "Hey ServiceNow, here is the server you need to connect to."
FieldValueWhy
TypeActive DirectorySelect the directory type
Server NameLDAP_ServerDisplay name — can be anything
Server URLldap.forumsys.com:389Host + port (389 for standard)
Starting Search DirDC=example,DC=comWhere ServiceNow begins searching
Login DNCN=read-only-admin,...Full DN path to login user
PasswordpasswordFrom the test server page
Login DN Format
The Login DN is NOT a simple username like "admin". In LDAP, usernames are written as a full path called a Distinguished Name (DN). Example: CN=read-only-admin,DC=example,DC=com
3
Configure Advanced Options
SETTINGS
These settings control how ServiceNow connects to and reads from the LDAP server.
OptionValueMeaning
Connection Timeout5 secondsHow long SN waits before giving up on connecting
Read Timeout10 secondsHow long SN waits to receive data back
SSL✓ (in production)ALWAYS enable in live org
ListenerON (recommended)AD pushes changes to SN automatically
Listener Interval5–60 minsHow often SN checks for new updates
✓ Listener Feature
When the Listener is ON, the moment a new user is created in Active Directory, ServiceNow is automatically notified and updated. No scheduled job needed — it's real-time!
4
Test the Connection
VERIFICATION
After saving the LDAP server record, always test the connection before moving forward.
How to test
  1. Save the LDAP server record
  2. Click "Test Connection" button on the form
  3. You should see: Connection successfully established
  4. If it fails → check the Server URL, port, Login DN, and password
05

LDAP OU Definition & Importing Users

After creating the server, you need to tell ServiceNow where inside the LDAP tree to look for users and groups. This is the LDAP OU Definition.

5
Configure the LDAP OU Definition (Users)
OU CONFIG
Once the server is saved, scroll down to the LDAP OU Definitions related list. Open "Users". This tells ServiceNow exactly which branch of the LDAP tree contains your users.
FieldValuePurpose
NameUsersLabel for this OU definition
RDNCN=usersWhere users are stored in the tree
Query FieldsAMAccountNameCRITICAL — unique ID field
Target Tablesys_userAuto-selected: ServiceNow Users table
ActivetrueEnable this OU definition
⚠ Query Field is Critical
The Query Field (sAMAccountName) tells ServiceNow which LDAP attribute to use to uniquely identify and search users. It stores the Windows login username (e.g., rSharma for Rahul Sharma).

If this field is wrong → users may not import properly, duplicates may be created, and authentication may fail.
6
Understanding the LDAP Filter
FILTERS
The filter controls exactly which users get imported. You can import all users, or filter by name, status, type, etc.
Default LDAP Filter — Decoded
(&(objectClass=person)(sn=*)(!(objectClass=computer))(!(userAccountControl:=2)))
objectClass=person
Only bring objects that are people/users. Do not bring printers, devices, or other objects.
sn=*
User must have a surname value (sn = surname). Ensures only valid user records with last names are included.
!(objectClass=computer)
Exclude computer objects. LDAP also stores machine and server records — we only want human users.
!(userAccountControl:=2)
Exclude disabled accounts. Only import active, enabled users into ServiceNow.
In plain English: "Bring only active human users with a surname. Ignore computers, printers, and disabled accounts."
Custom Filter Example (from the demo)
  1. To get only users whose name starts with E: (uid=E*)
  2. To get users starting with G: (uid=G*)
  3. In the OU Definition form → clear the default filter → type your custom filter → Save
  4. Click Browse to preview which users match
7
Load Records via Data Source
DATA IMPORT
ServiceNow uses a Data Source record to pull data from the LDAP server into a staging table, before the Transform Map maps it to the actual user table.
Steps
  1. In the OU Definition, scroll to the related list → click LDAP Server User (Data Source)
  2. Click Load All Records
  3. Wait for the job to complete
  4. Check: State = Complete, Completion Code = Success
  5. Verify counts: Process X, Inserts X, Updates 0, Errors 0
✓ In the Demo
After loading with filter uid=E* — Process: 3, Inserts: 3, Updates: 0, Errors: 0. Three users whose names start with E (including Einstein) were successfully loaded into the staging table.
06

Creating the Transform Map

The Transform Map tells ServiceNow: "This LDAP field maps to this ServiceNow field." It bridges the staging import table with the actual sys_user table.

8
Create & Configure the Transform Map
MAPPING
A Transform Map is how data from the staging import table gets written into the real ServiceNow table. You match each LDAP attribute to its corresponding ServiceNow field.
How to create it
  1. Navigate to System Import Sets → Transform Maps → New
  2. Set Target Table to sys_user (User table)
  3. Click Auto Map Matching Fields — SN will auto-match obvious fields
  4. Then click Map Assist to manually map remaining fields
  5. Save → Click Transform to run it

Field Mapping Reference

After running the transform, three users (including Einstein) appeared in the sys_user (User) table in ServiceNow. The LDAP data was fully mapped and imported successfully.
9
Automate with Scheduled Job
AUTOMATION
In production, you don't manually run the import every time. Create a Scheduled Job to automate it — daily, hourly, or at any defined interval.
How to set it up
  1. Navigate to System Definition → Scheduled Jobs → New
  2. Name it: LDAP User Sync - Daily
  3. Set the Run frequency — Daily, Hourly, Weekly, etc.
  4. In the script, reference the LDAP Data Source and trigger the import
  5. Save → Activate
💡 Or Use the Listener
If you enable the Listener on the LDAP Server record, changes in Active Directory push to ServiceNow automatically in real-time — no scheduled job needed for user updates.
